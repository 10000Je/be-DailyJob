# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습: Step-1 전체
  - 정적 index.html 응답 구현
  - HTTP Request 로그 출력 구현
  - Thread -> ThreadPool 개선
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: X

###### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: 파일을 바이트 배열로 변환하는데에 nio 패키지를 사용
- 그렇게 판단한 이유: 파일을 바이트 배열로 변환하는데에 nio 패키지를 쓰지 않고 구현하려면 보다 복잡한 코드를 작성하여야 하기 때문에 구현에 문제가 발생할 수 있었음
- 고려했지만 선택하지 않은 대안: ByteStream의 toByteArray() 메서드 사용
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 이미 구현된 클래스를 재사용하여 코드의 가독성과 재사용성을 높임

###### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: HTTP Request 로그 출력 시 사이트 로딩이 되지 않는 문제
- 처음 추측한 원인: logger.debug 메소드에 인자로 null인 String 객체를 넣으면 프로그램이 멈추는 것으로 추측함
- 원인을 확인하기 위해 시도한 방법: logger.debug 메소드에 null을 넣어서 출력해봄 -> 정상적으로 null 출력됨
- 실제 원인: BufferedReader의 readLine 메서드가 입력이 끝났음에도 null이 아닌 빈 문자열을 반환하여 탈출 조건(`line == null`)을 탈출하지 못함
- 해결 방법: 종료 조건을 `line == null || line.isEmpty()` 로 변경함
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: readLine()의 반환 값을 확인할 것

###### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: 문제 상황이 발생했을 때 먼저 반환 값부터 확인한 것
- 그 접근이 도움이 된 이유: 문제의 원인을 빠르게 찾을 수 있었음
- 다음에도 반복하고 싶은 점: 항상 반환 값을 확인하여 문제가 발생하는 시점을 찾는 것

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점: Step-1 의 목적이 "정적 index.html" 반환이여서 일단 index.html 파일의 내용만 반환하는 것을 목표로 두어야 했는데 이미지나 css가 제대로 로딩되지 않는 것을 보고 섣불리 모든 문제를 해결하려 한 것
- 그렇게 된 원인: 문제를 해결할 떄 계획을 제대로 세우지 않고 진행하려 한 것이 원인이라고 생각한다.
- 다음에는 다르게 해볼 구체적인 행동: 요구 사항(Specification)을 잘 읽어보고 현재 발생한 문제와 지금 해결해야하는 문제를 명확하게 정하고 행동할 것
