### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: 이전 PR이 Merge될 때까지 기다리지 않고 일단 fork한 리포지토리에 작업을 이어나간 것
- 그렇게 판단한 이유: Merge 될 떄까지 기다리면 구현 작업을 더이상 진행할 수 없었음
- 고려했지만 선택하지 않은 대안: 직접 upstream의 내 브랜치에 머지 작업을 진행하는 것
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 이후에 upstream 브랜치에 머지가 진행되었을때 내가 fork한 리포지토리에도 merge, rebase를 진행해줘야 해서 조금 번거로울 수 있음

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: 브라우저 상의 웹사이트가 이미지 요소들을 불러오지 못한 것
- 처음 추측한 원인: 파일 경로를 request로부터 파싱할 때 오류가 발생한다고 생각
- 원인을 확인하기 위해 시도한 방법: 파싱과정을 통해 반환된 값들을 계속 확인함
- 실제 원인: 클라이언트의 Accept 헤더의 첫번째 MIME 타입을 response의 content-type 헤더항목에 작성해주었는데
  이는 css 파일에선 정상작동 했지만 svg 이미지 파일에서는 올바르게 작동하지 못함
- 해결 방법: 클라이언트가 보내는 Accept 헤더가 아닌 리소스의 확장자를 통해서 직접 적절한 MIME 타입을 헤더에 작성하는 방향으로 수정하였음
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 정상 작동한 상황에서의 헤더와 이상 작동한 상황에서의 헤더를 제일 먼저 확인하는 것이 좋을 것 같다.
