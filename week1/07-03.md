### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: 회원가입 버튼을 눌러도 아무일도 일어나지 않는 증상
- 처음 추측한 원인: request를 처리하는 코드에 문제가 발생했다고 추측
- 원인을 확인하기 위해 시도한 방법: 소켓에 들어오는 HTTP Request Header 들을 확인
- 실제 원인: html이 잘못 작성되어 있어서 GET 요청 자체를 보내지 못한 것
- 해결 방법: html 파일을 적절히 수정
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 항상 로그를 제일 먼저 확인할 것

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: RequestHeader 클래스의 크기가 너무 커져서 이를 분리한 것
- 그 접근이 도움이 된 이유: RequestHeader 클래스가 전체적으로 줄어들어 코드를 읽기 쉽고 유지보수가 편해졌음
- 다음에도 반복하고 싶은 점: 구현할 때는 일단 기능의 완성에 중점을 두고, 이후에 리팩토링을 하는 것도 좋아보임

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점: HTTP Request 파싱 부분을 너무 대충 구현했다고 생각함
- 그렇게 된 원인: GET을 제외한 다른 요청을 고려하지 않았음.
- 다음에는 다르게 해볼 구체적인 행동: HTTP Request 파싱을 담당하는 모듈을 구현해볼 예정

# 주간 프롬프트 회고 템플릿

> 금요일 데일리 회고 템플릿 아래에 추가해서 작성하세요.

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트

원본 프롬프트: 지금 이 코드들을 JUnit과 AssertJ로 테스트하려면 어떻게 접근하는게 좋을까? 코드는 수정하지말고 일단 방법만 알려줘

선택한 이유: JUnit과 AssertJ로 테스트라는 구체적인 조건을 명시하였고, 코드 수정 없이라는 제약조건을 제시하여서 오류를 줄였으며
단순히 테스트 코드를 짜달라고 요청하는 것이 아닌 근본적인 접근 방법을 알고자 했기 때문

### 2. 원본 프롬프트 구조 분석

- 문제 상황: RequestHandler가 Socket과 같은 외부 의존성을 가지고 있어 테스트에 어려움을 겪는 상황
- 원하는 결과를 어떻게 요구했는가: 테스트 코드 결과물을 요구한게 아닌 접근 전략과 방법론에 대해 질문함
- 필요한 맥락이 포함되어 있는가: 지금 이 코드들 이라는 표현을 통해 RequestHandler.java가 테스트 대상임을 명확히함
- 판단 기준이나 제약 조건이 있는가: 코드는 수정하지 말고, 방법말 알려달라는 제약 조건을 정의
- 출력 방식이 명확한가: 방법만 알려달라는 것은 설명문을 필요로한다는 의미이므로 명확했음
- 나의 가설, 우려, 의도가 드러나는가: 단순히 동작하는 코드가 아닌 잘 테스트된 코드를 만드고자 하는 의도가 드러나있다.
  다만 프롬프트 자체에는 "Socket을 직접 생성해서 단위 테스트를 하기 어려울 것 같다" 라는 우려가 잘 드러나지 않음.

### 3. AI에게 프롬프트를 개선 요청하기

- 프롬프트 개선안

```
RequestHandler 클래스를 JUnit과 AssertJ로 테스트하고 싶습니다.
가장 큰 고민은 RequestHandler가 생성자에서 Socket 객체를 직접 주입받아 네트워크에 강하게 의존하고 있다는 점입니다. 이 때문에 실제 네트워크 연결 없이 순수 로직만을 분리하여 단위 테스트하기가 매우 까다로워 보입니다.
이런 강한 의존성을 가진 객체를 테스트하기 위한 표준적인 전략이나 디자인 패턴은 무엇인가요? 특히 Socket과 같은 외부 의존성을 테스트 중에 어떻게 처리(대체)하는지에 대한 접근법을 알려주세요.
제약 조건은 이전과 동일합니다. 1. 우선 RequestHandler의 원본 코드는 수정하지 않는다고 가정합니다. 2. 코드를 바로 작성하기보다는, Mockito와 같은 도구를 사용하는 것을 전제로 개념과 접근 방식 위주로 설명해주세요.
```
