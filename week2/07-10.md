# 주간 프롬프트 회고 템플릿

> 금요일 데일리 회고 템플릿 아래에 추가해서 작성하세요.

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트

원본 프롬프트:

```
궁금한건데 지금 클래스는 raw한 HTML 파일을 조립해서 응답으로 반환하는 방식으로 동적 HTML을 구현하고있는데 이러면 사용자 입력에 의해서 injection 공격이 발생할 수 있어. 이걸 막으려면 사용자의 입력이 HTML 문법이 아닌 순수한 텍스트로 인식되게 하고싶은데 방법이 없을까?
```

선택한 이유: 현재 상태를 정확히 서술하고 있고, 문제 원인을 스스로 생각 했으며, 어떻게가 아니라 목표 상태를 정의하였음.

### 2. 원본 프롬프트 구조 분석

- 문제 상황: 명확함. "raw HTML을 조립해 반환 → 사용자 입력으로 injection 발생 가능"이라고 원인-결과 구조로 서술. 증상이 아니라 메커니즘을 짚었음.
- 원하는 결과를 어떻게 요구했는가: 잘함. "고쳐줘"가 아니라 목표 상태("입력이 HTML 문법이 아닌 순수 텍스트로 인식")를 정의하고 "방법이 없을까?"로 열어둠. 결과는 고정, 수단은 개방 — 이상적 형태.
- 필요한 맥락이 포함되어 있는가: 부분적. 구현 방식("raw HTML 조립")은 있지만 어떤 클래스인지는 "지금 클래스"로만 지칭. 대화 흐름상 통했지만, 맥락 없이 이 프롬프트만 보면 대상 특정 불가. ResponseHandler처럼 클래스명이 있었으면 더 자족적.
- 판단 기준이나 제약 조건이 있는가: 없음. "표준 라이브러리로", "외부 의존성 없이", "성능 고려" 같은 제약이 빠짐. 실제로 다음 턴에 "Java 표준 라이브러리에 이미 있어?"를 따로 물어야 했음 → 이 제약을 처음에 넣었으면 왕복 하나 줄었음.
- 출력 방식이 명확한가: 불명확. "방법이 없을까?"라 설명/코드/예시 중 무엇을 원하는지 안 정함. 결과적으로 개념 설명부터 시작해 잘 흘러갔지만, 의도된 게 아니라 운이 좋았던 것.
- 나의 가설, 우려, 의도가 드러나는가: 강점. 우려(injection 위험)와 의도(순수 텍스트 처리)가 뚜렷. 다만 "이럴 것 같다"는 가설은 없음 — 예: "이스케이프 처리하면 될 것 같은데 맞아?"처럼 자기 추측을 얹었으면 검증형 대화로 더 깊어졌을 것.

### 3. AI에게 프롬프트를 개선 요청하기

- 프롬프트 개선안

```
지금 ResponseHandler가 {{변수}} 자리에 사용자 입력을 그대로 끼워 raw HTML을 조립해서 반환하고 있어. 이러면 사용자가 <script> 같은 태그를 입력했을 때 그대로 실행되는 XSS가 가능하다고 판단했어.

입력이 HTML 문법이 아니라 순수 텍스트로만 인식되게 막고 싶어. 내 생각엔 특수문자를 이스케이프하는 방향일 것 같은데 맞아?

제약은 외부 라이브러리 없이 표준 API 범위 안에서 하는 거야. 먼저 접근 방법과 원리를 개념 위주로 설명해줘. 코드는 아직 필요 없어.
```

### 4. AI가 개선한 프롬프트 분석

- 예) 원본 프롬프트에 비해- 맥락/ 더 명확해진 목표/ 추가된 제약 조건/ 추가된 출력 형식/ 답변 품질이 좋아질 것으로 예상되는 이유 등

```
기준: 추가한 것
맥락: ResponseHandler, {{변수}} 치환 지점을 특정 → 프롬프트만으로 대상 파악 가능
문제 상황: <script> 입력이라는 구체 예시로 위험을 재현 가능하게
가설: "이스케이프 방향일 것 같은데 맞아?" → 검증형 질문으로
제약 조건: "외부 라이브러리 없이 표준 API" 명시 → 왕복 하나 절약
출력 방식: "개념 위주 설명, 코드는 아직" → 응답 형태 고정
원하는 결과: 목표 상태(순수 텍스트 인식)는 원본 그대로 유지
```
