# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습: 댓글 기능 구현(작성자 프로필 이미지 포함), 닉네임/비밀번호 변경 기능,
  좋아요/댓글 수 기능, 로그인 실패 시 알림 UX 개선, 인메모리 Database를 H2 + JDBC 기반으로 전환,
  WAS 프레임워크 코드와 애플리케이션 코드 분리 리팩토링, 리팩토링에 맞춘 테스트 코드 전면 정비 및 신규 테스트 추가
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: Database 예외를 RuntimeException 대신 명확한 타입으로 던지고
  핸들러에서 받아 적절한 응답으로 처리하는 예외 구조 개선(문제 인지만 하고 추후 리팩토링으로 미룸),
  Comment 이후 나머지 신규 테스트 작성

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: WAS 프레임워크와 애플리케이션 코드를 분리하면서, WebServer를 싱글턴으로 만들지 않고 생성자 기반 생성을 유지한 결정
- 그렇게 판단한 이유: 한 애플리케이션에서 여러 서버를 띄우는 것도 가능해야 하는데 싱글턴은 그 확장성을 막기 때문
- 고려했지만 선택하지 않은 대안: WebServer를 싱글턴 패턴으로 얻는 방식
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 전역 상태를 줄이는 방향과 일관되게 감. 같은 이유로 Router도
  전역 상태에서 인스턴스 기반 + 주입 사슬 구조로 바꿔 테스트 격리(DB 격리 포함)가 가능해짐

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: IntelliJ에서 WebServer를 실행하면 DB는 정상 동작하는데 H2 웹 콘솔(localhost:8082)에는 접속이 안 됨
- 처음 추측한 원인: 콘솔 URL이나 설정을 잘못 입력했다고 추측
- 원인을 확인하기 위해 시도한 방법: 콘솔 접속을 여러 방식으로 재시도하고, H2 웹 콘솔이 어떻게 동작하는지
  (별도 프로그램이 어떻게 내 프로그램의 DB에 접근하는지) 원리부터 질문해서 확인
- 실제 원인: H2 웹 콘솔은 WAS와 별개로 띄워야 하는 독립 서버라서, WebServer만 실행하면 콘솔 서버 자체가 떠 있지 않음
- 해결 방법: WebServer 기동 시 H2 웹 콘솔 서버를 함께 띄우도록 하고, 파일 기반 DB에 AUTO_SERVER=TRUE를 붙여 동시 접근 가능하게 함
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 접속하려는 대상이 내 프로그램에 내장된 것인지 별도로 띄워야 하는 프로세스인지, 그 서버가 실제로 떠 있는지부터 확인할 것

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: 요구사항에 없는 문제를 스스로 미리 짚은 것 — 좋아요 증가 시 멀티스레드 race condition을
  구현 전에 지적해 AtomicLong으로 처리했고, 닉네임 변경 시 중복 검사가 필요하다는 것도 스스로 도출함
- 그 접근이 도움이 된 이유: 기능이 "일단 동작"한 뒤에는 드러나지 않는 동시성/정합성 버그를 구현 단계에서 차단할 수 있었음
- 다음에도 반복하고 싶은 점: 기능 구현 전에 동시성, 중복, 경계 조건 같은 암묵적 요구사항을 먼저 점검하는 습관

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점: likeCount를 commentCount로 착각해서 "Article에 포함하지 않는 방향"으로
  잘못된 요청을 내렸다가 뒤늦게 정정해서 재작업한 것. 또 리팩토링을 다 끝낸 뒤에야 테스트 코드를 한꺼번에 몰아서 정비한 것
- 그렇게 된 원인: 비슷한 이름의 필드를 혼동한 채 결정을 내렸고, 구현 속도에 집중하느라 테스트를 변경과 함께 따라가게 하지 않음
- 다음에는 다르게 해볼 구체적인 행동: 설계 결정을 내리기 전에 대상 필드/클래스 이름을 정확히 확인하고 말하기,
  구조 변경 커밋마다 관련 테스트를 같이 수정해서 테스트 부채가 쌓이지 않게 하기
