# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습: Database 예외 구조 개선(RuntimeException 대신 SQLException을 던져
  핸들러에서 예외 처리를 강제, 예외 메시지 상수화), 핸들러에서 서비스 로직 분리(UserService·LoginService),
  세션 검증-획득을 Optional 반환 메소드 하나로 통합, RequestParser 파싱 실패 예외 처리 정리,
  MultiFormRequestBody의 getFile에 Optional 적용, 모든 public 메소드에 자바독 컨벤션 정해서 주석 작성
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: Part 내부 getter(getFilename, getContentType)의 Optional 적용
  (범위가 커져서 별도 작업으로 미룸), 핸들러가 핸들러를 직접 호출하는 구조(로그아웃 처리) 정리

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: 핸들러가 HttpRequest 처리부터 서비스 로직, HttpResponse 반환까지 다 하던 구조에서
  서비스 로직을 별도 계층(UserService 등)으로 분리하기로 한 결정. 이때 로그인 확인은 DB를 물지 않고
  세션 계층에서만 처리하도록 경계를 정함
- 그렇게 판단한 이유: 핸들러는 HTTP 요청/응답 변환만 담당하고 도메인 로직은 서비스가 담당해야
  책임이 명확해지고, 세션 계층이 DB에 의존하지 않아야 계층 간 경계가 깨끗해지기 때문
- 고려했지만 선택하지 않은 대안: 지금처럼 핸들러 안에서 서비스 로직까지 함께 처리하는 구조 유지
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 핸들러가 얇아져 역할이 분명해졌고,
  로그인 검증 로직이 한 군데로 모여 중복이 사라짐

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: 핸들러에서 isLoggedIn 체크를 통과하면 getSession이 항상 유효한데도,
  IDE가 두 번째 호출에서 "비어있을 수 있다"고 경고를 계속 띄움
- 처음 추측한 원인: 중복 체크를 안 해서 생기는 단순 IDE 경고라고 생각했지만, 검증 로직이
  이렇게 분리된 설계 자체가 마음에 안 든다는 위화감이 있었음
- 원인을 확인하기 위해 시도한 방법: 경고를 억지로 없애는 대신 "검증이 왜 이렇게 분리되어 있는가"를
  설계 관점에서 다시 질문하고 논의함
- 실제 원인: "검증(isLoggedIn)"과 "값 획득(getSession)"이 두 번의 호출로 쪼개져 있어서,
  첫 호출에서 알아낸 "유효하다"는 사실이 타입에 담기지 않아 두 번째 호출 때 타입 시스템이 그걸 모르는 것
- 해결 방법: 세션 계층에 검증과 획득을 한 번에 하는 Optional(userId) 반환 메소드를 추가하고,
  isLoggedIn은 그 메소드 기반으로 재구현해 검증 로직을 한 군데로 모음
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: IDE 경고가 반복되면 경고 억제가 아니라
  "코드가 아는 사실이 타입에 담겨 있는가"를 먼저 볼 것. 검증과 값 획득은 한 호출로 합칠 수 있는지 확인

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: ArticleHandler에서 "뭔가 맘에 걸린다"는 위화감을 그냥 넘기지 않고 코드 점검을 요청한 것
- 그 접근이 도움이 된 이유: 점검 결과 Get에만 새 패턴이 적용되고 Post/Like는 옛 패턴이 남은 불일치,
  Optional 미반영으로 컴파일이 깨지는 곳, handleLikePost에 로그인 체크가 아예 빠진 것까지 발견해서
  잠재 버그를 미리 잡을 수 있었음. 어제 미뤄뒀던 예외 구조 개선을 오늘 바로 처리한 것도 부채를 쌓지 않는 데 도움이 됨
- 다음에도 반복하고 싶은 점: 코드에 위화감이 들면 이유를 설명할 수 없어도 일단 점검하고 넘어가기,
  미룬 작업은 다음 날 최우선으로 처리하기

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점: Optional 전환과 로그인 체크 통일 같은 리팩토링을
  일부 메소드(Get)에만 적용하고 Post/Like 등 나머지 호출부에는 반영하지 않아, 컴파일 에러와
  패턴 불일치가 생긴 채로 한동안 진행한 것
- 그렇게 된 원인: 반환 타입을 바꾸면 그 여파가 모든 호출부에 미치는데, 변경 지점만 보고
  호출부 전체를 추적하지 않음
- 다음에는 다르게 해볼 구체적인 행동: 메소드 시그니쳐(특히 반환 타입)를 바꾸면 그 자리에서
  호출부를 전부 찾아 같이 고치고, 컴파일 확인까지 마친 뒤에 다음 작업으로 넘어가기
