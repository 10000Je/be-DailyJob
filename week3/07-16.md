# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습: 입력 제한·예외 처리 체계 구축(step8, PR 머지).
  HttpStatus·HttpMethod enum과 HttpException 계층(400/413/414/431/501/500) 도입,
  라우터 키를 path 기준으로 재편해 잘못된 메소드는 405+Allow, 미지원 메소드는 501로 응답,
  RequestLimits로 요청 라인·헤더·바디 크기 상한 도입(앱단에서 세터로 조정 가능),
  Validator 인터페이스를 만들어 핸들러에 흩어진 형식 검증을 이관하고 중복 응답을 409로 정정,
  HTML 폼에 클라이언트 1차 입력 제한 추가. DB 예외를 전용 DatabaseException으로 교체,
  글 이미지를 실제 확장자로 저장하고 파일-DB 저장 일관성 보장. 1GB 업로드·초대형 헤더를
  curl로 직접 날려 방어가 동작하는지 검증
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 프로필 이미지 확장자 처리(구현까지 했다가
  적용하지 않기로 하고 되돌림, 후속 작업으로 남김), 핸들러 멀티파트 테스트(테스트 DB 초기화
  문제를 먼저 손봐야 함), 테스트용으로 임시 전환한 인메모리 DB 원복

### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: 예외 체계를 만들 때 부모 HttpException이 HttpStatus enum을 필드로 갖고,
  자식 예외(BadRequestException 등)는 타입별로 상태코드를 고정하도록 설계한 것
- 그렇게 판단한 이유: 상태코드를 자유롭게 넣을 수 있으면 InternalServerException에
  500번대가 아닌 코드를 넣고 던지는 오용이 가능한데, enum과 타입 고정으로 이를
  컴파일 타임에 차단할 수 있기 때문
- 고려했지만 선택하지 않은 대안: 상태코드를 int로 받는 단일 예외 클래스(유연하지만
  잘못된 코드가 들어가는 걸 막을 수 없음), checked 예외 사용
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 핸들러는 의미에 맞는 예외만 던지고
  RequestHandler의 공통 처리 지점이 상태코드 변환을 전담하게 되어, 예외→응답 매핑이
  한 군데로 모이고 잘못된 상태코드가 나갈 경로가 타입 차원에서 사라짐

### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: 이미지 저장이 실패해도 사용자에게는 302 리다이렉트가 나가서
  글이 정상 등록된 것처럼 보이고, DB에는 이미지 없는 고아 글이 남음
- 처음 추측한 원인: 이미지 저장 과정에서 예외가 그대로 삼켜지고 있다는 의심
- 원인을 확인하기 위해 시도한 방법: "핸들러에서 예외가 발생하면 어떻게 처리되는가"를
  먼저 확인하고, saveImage 주변 흐름과 글 INSERT 순서를 짚어봄
- 실제 원인: saveImage가 예외를 잡아 error 로그만 남기고 삼키는 것 + "글 INSERT 후
  이미지 저장" 순서라서 저장이 실패하면 DB에 글만 남는 구조. 파일시스템과 DB라는
  서로 다른 자원을 묶는 트랜잭션이 없어 일관성이 수동으로 관리돼야 했음
- 해결 방법: 예외를 삼키지 않고 전파해 500 안전망으로 흘려보내고, 순서를 "이미지 먼저
  저장 → 성공하면 경로까지 담아 INSERT"로 뒤집어 고아 글을 차단. DB 쓰기가 실패하면
  저장한 파일을 지우는 보상 처리를 넣고, 이미지 경로 자체를 DB에 저장하도록 변경
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 예외를 catch해서 로그만 남기고 삼키는
  지점이 없는지, 서로 다른 자원(파일/DB)에 쓰는 작업은 "앞이 실패하면", "뒤가 실패하면"
  두 방향 모두 어떤 상태가 남는지 점검할 것

### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: 바디 크기 제한을 만들다가 "악의적인 클라이언트가 헤더만 엄청 길게
  보내면 어떻게 될까?"라는 공격 시나리오를 스스로 떠올려 헤더 크기·개수 제한(431)까지

  보내봐서 방어가 진짜로 작동함을 확인함
- 다음에도 반복하고 싶은 점: 방어 로직을 만들 때 "이걸 우회하려면 어떻게 공격할까"를
  한 번 더 묻기, 방어 코드는 실제 공격 요청을 만들어 직접 검증하고 끝내기

# 주간 프롬프트 회고 템플릿

> 금요일 데일리 회고 템플릿 아래에 추가해서 작성하세요.

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트

원본 프롬프트:

```
예외 전파하도록 수정하는데 꼭 짚어야 할게 하나 더 있어.
"이미지 저장이 실패"하면 데이터베이스에 저장하는 것도 해선 안돼.
그리고 이미지 저장이 성공하더라도 데이터베이스에 쓰는 작업을 실패하면 이미지를 저장해선 안되고 500 응답을 제대로 던져주도록 해야해
```

선택한 이유: "예외를 전파해줘"라는 단순 수정 요청에서 멈추지 않고, 파일과 DB라는 서로 다른
자원 사이에서 실패가 일어날 수 있는 두 방향을 모두 짚어 "어떤 상태가 남으면 안 되는지"를
불변 조건으로 정의했음. 수단(순서 뒤집기, 보상 삭제 등)은 열어두고 결과 조건만 고정해서,
AI가 설계로 답할 수 있게 만든 프롬프트였음.

### 2. 원본 프롬프트 구조 분석

- 문제 상황: 부분적. "예외 전파하도록 수정하는데"로 진행 중인 작업 맥락은 얹었지만,
  어떤 핸들러의 어떤 메소드인지, 현재 저장 순서가 어떤지는 대화 흐름에 의존.
  이 프롬프트만 떼어놓고 보면 대상을 특정할 수 없음.
- 원하는 결과를 어떻게 요구했는가: 강점. "이렇게 구현해줘"가 아니라 실패 시나리오별로
  도달하면 안 되는 상태(글만 남음/이미지만 남음)와 보장해야 할 응답(500)을 조건으로
  정의함. 결과는 고정하고 수단은 개방한 형태.
- 필요한 맥락이 포함되어 있는가: 부족. saveImage가 예외를 삼킨다는 사실, "글 INSERT 후
  이미지 저장" 순서라는 현재 구조가 빠져 있음. 직전 대화에서 공유된 상태라 통했지만
  자족적이지는 않음.
- 판단 기준이나 제약 조건이 있는가: 강점. 두 실패 방향 각각의 금지 조건 + 500 응답이라는
  판단 기준이 명확해서, 나온 설계안을 이 기준으로 바로 검증할 수 있었음.
- 출력 방식이 명확한가: 없음. 설계 논의를 원하는지 바로 구현을 원하는지 안 정함.
  실제로는 설계 설명부터 흘러가서 좋았지만 의도한 결과는 아니었음.
- 나의 가설, 우려, 의도가 드러나는가: 우려(불일치 상태가 남는 것)는 뚜렷하지만,
  "순서를 바꾸면 될 것 같다" 같은 자기 가설은 없음. 가설을 얹었으면 검증형 대화로
  더 빨리 수렴했을 것.

### 3. AI에게 프롬프트를 개선 요청하기

- 프롬프트 개선안

```
ArticleHandler의 글 작성 흐름이 지금 "글 INSERT → saveImage" 순서인데, saveImage가 예외를 잡아 로그만 남기고 삼키고 있어. 그래서 이미지 저장이 실패해도 302가 나가고 이미지 없는 글만 DB에 남아.

예외를 전파하도록 고치려는데, 파일시스템과 DB는 공용 트랜잭션이 없어서 일관성을 수동으로 보장해야 한다고 판단했어. 지켜야 할 조건은 두 가지야:

1. 이미지 저장이 실패하면 DB에 글이 남아선 안 됨
2. 이미지 저장이 성공해도 DB 쓰기가 실패하면 이미지 파일이 남아선 안 되고, 500 응답이 나가야 함

내 생각엔 저장 순서를 뒤집고 실패 시 보상 처리를 넣는 방향일 것 같은데 맞아? 코드는 아직 필요 없고, 각 실패 시나리오에서 어떤 상태가 남는지를 기준으로 설계안을 먼저 설명해줘.
```

### 4. AI가 개선한 프롬프트 분석

- 예) 원본 프롬프트에 비해- 맥락/ 더 명확해진 목표/ 추가된 제약 조건/ 추가된 출력 형식/ 답변 품질이 좋아질 것으로 예상되는 이유 등

```
기준: 추가한 것
맥락: ArticleHandler, "글 INSERT → saveImage" 순서, 예외 삼킴 지점을 특정 → 프롬프트만으로 현재 구조 파악 가능
문제 상황: "302가 나가고 이미지 없는 글만 남는다"는 관찰된 증상을 명시 → 문제를 재현 가능하게
판단 근거: "공용 트랜잭션이 없어 수동 보장 필요"라는 문제 인식을 드러냄 → AI가 같은 프레임에서 답하게 됨
가설: "순서 뒤집기 + 보상 처리 방향일 것 같은데 맞아?" → 검증형 질문으로 전환
출력 방식: "코드는 아직 필요 없고 설계안 먼저" → 응답 형태 고정, 구현으로 건너뛰는 것 방지
검증 기준: "각 실패 시나리오에서 남는 상태"를 설명 기준으로 지정 → 답변을 불변 조건으로 바로 검증 가능
원하는 결과: 두 실패 방향의 불변 조건은 원본의 강점 그대로 유지
```
